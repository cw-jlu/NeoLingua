## netty架构

### 1. meeting_service

#### 原因
- 需要处理多人实时音频消息
- 高并发场景（多个房间同时进行）
- 低延迟要求（实时语音交互）

#### 技术实现
- **Netty 版本**: 通过 Spring Boot 3.2.0 自动管理
- **WebSocket 端口**: 8090（可配置）
- **线程模型**: 
  - Boss 线程: 1
  - Worker 线程: 自动（CPU 核心数 × 2）
- **消息格式**: JSON
- **心跳机制**: 60秒读超时


#### Netty 配置规范

##### application.yml
```yaml
netty:
  websocket:
    port: 8090              # WebSocket 端口
    boss-threads: 1         # Boss 线程数
    worker-threads: 0       # Worker 线程数（0=自动）
    max-frame-size: 65536   # 最大消息大小（字节）
    read-timeout: 60        # 读超时（秒）
```

##### 线程配置建议
- **Boss 线程**: 通常 1 个即可
- **Worker 线程**: 
  - 默认: CPU 核心数 × 2
  - 高并发: CPU 核心数 × 4
  - 低延迟: CPU 核心数 × 1

##### 内存配置建议
- **最大帧大小**: 
  - 文本消息: 4KB - 16KB
  - 音频消息: 64KB - 256KB
  - 大文件: 1MB - 10MB

#### 线程安全
- 使用 `ConcurrentHashMap` 管理连接
- 避免共享可变状态
- 使用 `@ChannelHandler.Sharable` 标记

#### 资源释放
```java
@Override
public void channelInactive(ChannelHandlerContext ctx) {
    // 清理资源
    String channelId = ctx.channel().id().asLongText();
    CHANNEL_MAP.remove(channelId);
}
```

### 常见问题

#### 1. 端口冲突
**现象**: 服务启动失败，提示端口被占用

**解决方案**:
```bash
# Windows
netstat -ano | findstr :8090
taskkill /PID <PID> /F

# Linux
lsof -i :8090
kill -9 <PID>
```

#### 2. 连接超时
**现象**: 客户端连接后很快断开

**排查步骤**:
1. 检查心跳配置
2. 检查客户端是否发送心跳
3. 调整 read-timeout 配置
4. 检查网络稳定性

#### 3. 消息丢失
**现象**: 部分消息未收到

**排查步骤**:
1. 检查消息大小是否超过限制
2. 检查网络稳定性
3. 检查服务端日志
4. 检查消息序列化

#### 4. 内存泄漏
**现象**: 内存持续增长

**排查步骤**:
1. 检查连接是否正确释放
2. 检查 ByteBuf 是否正确释放
3. 使用 JVM 工具分析堆内存
4. 检查业务代码是否有内存泄漏
